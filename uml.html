📐 Qu'est-ce que l'UML ? (Rappel Rapide)

Définition Officielle

L'UML (Unified Modeling Language) est un langage de modélisation graphique standardisé utilisé en génie logiciel pour visualiser, spécifier, construire et documenter les systèmes informatiques.

  • Standard : Norme OMG (Object Management Group)
  • Version actuelle : UML 2.5
  • 14 diagrammes répartis en 2 catégories
  • Indépendant : de tout langage de programmation

Pourquoi UML dans les Examens OFPPT ?

  • Analyse et conception : Étape préalable au développement
  • Communication : Langage commun entre développeurs et clients
  • Génération de code : Outils comme StarUML, Enterprise Architect
  • Documentation : Structure visuelle claire pour la maintenance
  • Teste la logique : Compréhension des relations entre objets

👤 DIAGRAMME DE CAS D'UTILISATION (Use Case)

👤 Le diagramme le plus demande dans les examens OFPPT

Le diagramme de cas d'utilisation représente les fonctionnalités du système du point de vue de l'utilisateur.

📋 Éléments de Notation

Élément Représentation Description
Acteur 🧑 Stickman (ou rectangle <>) Personne, système ou entité qui interagit avec le système
Use Case 🥚 Ellipse (ovale) Fonctionnalité ou service offert par le système
Système (Boundary) 📦 Rectangle (cadre du système) Limite du système, contient les use cases
Association ➡️ Ligne simple Relie un acteur à un use case
Include ➡️ Flèche pointillée avec <> Un use case inclut obligatoirement un autre
Extend ➡️ Flèche pointillée avec <> Un use case étend optionnellement un autre
Généralisation ⬆️ Flèche creuse Relation d'héritage entre acteurs ou use cases

🏛️ DIAGRAMME DE CLASSES

🏛️ Le diagramme structurel le plus important

Le diagramme de classes modélise la structure statique du système : les classes, leurs attributs, méthodes et les relations entre elles.

📋 Représentation d'une Classe

Notation d'une Classe UML
┌─────────────────────────────┐ │ NomClasse │ ← Nom de la classe ├─────────────────────────────┤ │ - attributPrive: Type │ ← Attributs (visibilité + nom + type) │ # attributProtege: Type │ │ + attributPublic: Type │ ├─────────────────────────────┤ │ + méthodePublique(): Type │ ← Méthodes │ - méthodePrivée(): void │ └─────────────────────────────┘ Visibilité : + = public (accessible par tous) - = private (accessible uniquement dans la classe) # = protected (accessible dans la classe et sous-classes) ~ = package (accessible dans le même paquetage)

🔗 Les Relations entre Classes

Relation Notation Description Exemple
Association ➖ Ligne simple Relation structurelle entre classes Employé → Entreprise
Agrégation ◇── Ligne avec losange vide Relation "a un" (partie faible) Voiture ◇── Roue
Composition ◆── Ligne avec losange plein Relation "est composé de" (partie forte) Maison ◆── Pièce

⏱️ DIAGRAMME DE SÉQUENCE

⏱️ Montre l'interaction temporelle entre objets

Le diagramme de séquence illustre l'ordre chronologique des messages échangés entre objets pour réaliser un scénario.

📋 Éléments de Notation

Élément Représentation Description
Ligne de vie ⬇️ Ligne pointillée verticale Représente la durée de vie d'un objet
Acteur/Objet 📋 Rectangle en haut de la ligne Entité qui participe à l'interaction
Message synchrone ➡️ Flèche pleine → Appel bloquant (attente de réponse)
Message asynchrone ➡️ Flèche avec trait →| Appel non bloquant
Message retour ⬅️ Flèche pointillée <-- Valeur de retour d'un message
Fragment combiné 📐 Rectangle avec opérateur (alt, opt, loop) Condition, option, boucle
Occurrence d'exécution ▯ Rectangle fin sur la ligne de vie Période d'exécution d'une opération

📋 Opérateurs de Fragments Combinés

Opérateur Signification

optOptionnel (facultatif)
loopBoucle (itération)
parParallèle (exécution concurrente)

⚡ DIAGRAMME D'ACTIVITÉ

⚡ Modélise les flux de travail et processus métier

Le diagramme d'activité est similaire à un organigramme. Il montre le flux de contrôle d'une activité à une autre.

📋 Éléments de Notation

Élément Représentation Description
Nœud initial ● Cercle plein Point de départ du processus
Action ▭ Rectangle aux coins arrondis Étape du processus
Décision ◇ Losange Branchement conditionnel
Fork ━ Barre épaisse (entrante 1 → sortante N) Démarrage de flux parallèles
Join ━ Barre épaisse (entrante N → sortante 1) Synchronisation de flux parallèles
Nœud final ◎ Cercle avec bordure épaisse Fin du processus
Partition (Swimlane) ▭ Rectangle vertical/horizontal Regroupe les actions par acteur responsable
Flux ➡️ Flèche Transition entre actions

📋 Exemple de Diagramme d'Activité

Diagramme d'Activité - Processus de Commande
● Début ↓ ▭ Passer commande ↓ ◇ Stock disponible ? ├───Oui───→ ▭ Préparer colis │ ↓ │ ▭ Expédier commande │ ↓ │ ┌─────────────────┐ │ │ ▭ Envoyer │ │ │ notification │ ← Parallèle (Fork) │ │ ▭ Mettre à │ │ │ jour stock │ │ └────────┬────────┘ │ ↓ (Join) ├───Non────→ ▭ Annuler commande │ ↓ │ ▭ Notifier client ↓ ↓ ◎ Fin ◎ Fin

🔄 DIAGRAMME D'ÉTATS (Statechart)

🔄 Modélise le cycle de vie d'un objet

Le diagramme d'états montre les différents états par lesquels passe un objet et les transitions entre ces états déclenchées par des événements.

📋 Éléments de Notation

Élément Représentation Description
État ▭ Rectangle aux coins arrondis Situation dans le cycle de vie
État initial ● Cercle plein État de départ
État final ◎ Cercle avec bordure État de terminaison
Transition ➡️ Flèche Changement d'état déclenché par un événement
Événement Étiquette sur la transition Déclencheur du changement d'état
État composite ▭ Avec sous-états imbriqués État contenant d'autres états

✅ RÉPONSE CORRECTE

UML (Unified Modeling Language) est un langage de modélisation graphique standardisé utilisé en génie logiciel pour visualiser, spécifier, construire et documenter les systèmes informatiques.

Les 2 catégories de diagrammes UML :

  • Diagrammes structurels (statiques) : Décrivent la structure du système
    Exemples : Diagramme de classes, Diagramme d'objets, Diagramme de composants
  • Diagrammes comportementaux (dynamiques) : Décrivent le comportement du système
    Exemples : Diagramme de cas d'utilisation, Diagramme de séquence, Diagramme d'activité

RelationNotationExplicationExemple

Question 2 3 points

Énoncé : Expliquer le concept de "généralisation" (héritage) en UML. Donner un exemple.

✅ RÉPONSE CORRECTE

La généralisation (ou héritage) est une relation entre une classe mère (super-classe) et une classe fille (sous-classe) où la sous-classe hérite des attributs et méthodes de la super-classe.

Notation : Flèche creuse de la sous-classe vers la super-classe.

Exemple :

┌──────────────┐ │ Véhicule │ ← Super-classe ├──────────────┤ │ - vitesse │ │ - couleur │ ├──────────────┤ │ + demarer() │ │ + arreter() │ └──────┬───────┘ ↑ généralisation ┌──────┴───────┐ ┌──────────────┐ │ Voiture │ │ Moto │ ← Sous-classes ├──────────────┤ ├──────────────┤ │ + nbPortes │ │ + aSidecar │ └──────────────┘ └──────────────┘
Question 3 3 points

Énoncé : Citer les opérateurs de fragments combinés dans un diagramme de séquence et expliquer leur rôle.

✅ RÉPONSE CORRECTE
OpérateurRôle
altAlternatives conditionnelles (if/else). Plusieurs options mutuellement exclusives.
optOptionnel. Le fragment s'exécute si la condition est vraie.
loopBoucle. Répète le fragment tant que la condition est vraie.
parParallèle. Les interactions dans le fragment s'exécutent en parallèle.
refRéférence. Renvoie vers un autre diagramme de séquence.
breakSortie conditionnelle. Interrompt le fragment si la condition est vraie.

❌ ERREURS FRÉQUENTES EN UML

💀 ERREUR #1 : Confondre Agrégation et Composition
❌ FAUX : "Agrégation et composition sont identiques."

✅ CORRECT : "L'agrégation (◇) est une relation faible, le composant peut exister sans le conteneur. La composition (◆) est une relation forte, le composant ne peut pas exister sans le conteneur."
💀 ERREUR #2 : Inverser include et extend
❌ FAUX : "<> est optionnel et <> est obligatoire."

✅ CORRECT : "<> est obligatoire (le use case de base inclut toujours le use case inclus). <> est optionnel (le use case d'extension est facultatif)."
💀 ERREUR #3 : Oublier les multiplicités
❌ FAUX : "Les multiplicités sont optionnelles dans un diagramme de classes."

✅ CORRECT : "Les multiplicités sont obligatoires pour définir précisément les relations. Sans elles, le modèle est ambigu."
💀 ERREUR #4 : Diagramme d'Activité vs Organigramme
❌ FAUX : "Un diagramme d'activité est juste un organigramme."

✅ CORRECT : "Un diagramme d'activité UML supporte des concepts avancés comme les swimlanes (partitions), les flux parallèles (fork/join), les signaux et les événements."

📋 Tableau Récapitulatif des Diagrammes UML

Diagramme Catégorie Objectif Éléments Clés
Cas d'Utilisation Comportemental Fonctionnalités du système vues par l'utilisateur Acteur, Use Case, Boundary, <>, <>
Classes Structurel Structure statique du système Classe, Attribut, Méthode, Association, Héritage
Séquence Comportemental Interactions temporelles entre objets Ligne de vie, Message, Fragment (alt/opt/loop)
Activité Comportemental Flux de travail et processus métier Action, Décision, Fork/Join, Swimlane
États Comportemental Cycle de vie d'un objet État, Transition, Événement, État composite
